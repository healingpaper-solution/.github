# KOS Branch / Commit 전략

## 기본 원칙
- 작게 쪼개고 자주 통합해요.
- KOS는 기본적으로 `main` 브랜치를 기준으로 브랜치를 생성하여 작업해요.

## 브랜치 네이밍
- TASK ID가 있는 경우: `{my_nickname}/{task_id}` 형식을 사용해요.
- TASK ID가 없는 경우: `{my_nickname}/{kebab-case}` 형식을 사용해요.

## 커밋 규칙
- TASK ID가 있는 경우:
  - 커밋 메시지는 반드시 `[{task_id}]`로 시작해요.
  - 커밋 메시지는 한국어 문장으로 작성해요. (~한다.)
    - 예: `[KT-2662] GetZonesQueryProcessor에 isDefault 응답을 추가한다.`
- TASK ID가 없는 경우:
  - `[TASK ID]`를 포함하지 않아요.
  - 동일하게 한국어 문장으로 작성해요.

## 배포 전략
- KOS 제품은 `main` 기준으로 tag를 생성하고 배포해요.
- Trunk-based Deployment를 기본으로 사용해요.
- 그렇기 때문에, 모든 변경 사항은 지속적 통합이 고려되어야만 합니다. 맥락 없는 사람이 main을 기준으로 배포했을 때 장애가 나거나, 문제가 날 수 있는 커밋이 포함되어선 안 됩니다. '배포 가능 상태 유지' 를 확인하세요.

## 병합 전략
- Sub Branch는 `rebase and commit` 전략으로 병합해요.
- 충돌이 발생하면 rebase로 해결해요.

# PR 가이드

## PR 크기
- 변경 라인은 300줄 이하를 지향해요.
- 테스트 코드는 라인 수에서 제외해요.

## 변경 범위
- 하나의 PR에는 하나의 컨텍스트만 포함해요.

❌ 잘못된 예시
```
PR#999 [KT-9999] IDL 최신화하고 XXXRepository와 XXXUsecase를 작성해요
- IDL 최신화
- Repository 구현
- Usecase 구현
```

## 배포 가능 상태 유지
- 모든 커밋은 main에 병합되어도 애플리케이션이 깨지지 않아야 해요.
- 이를 위해 Feature Flag 사용을 권장해요.

## 승인 규칙
- 최소 1명 이상의 approve를 받아야 병합할 수 있어요.

## 코멘트 반영 방식
- 새로운 커밋을 추가하지 않고 `git commit --amend`로 수정해요.

# Q&A

## Q. 왜 merge, squash merge가 아니라 rebase merge를 사용하나요?

### A. 선형적인 히스토리를 유지하기 위해서예요.

- Linear History의 장점:
  - 특정 변경 사항을 되돌리기 쉬워요.
  - 히스토리가 단순해서 추적과 디버깅이 쉬워요.

---

## Q. 왜 코멘트 반영 시 amend를 사용하나요?

### A. 커밋 단위를 깨지 않기 위해서예요.

“코멘트 반영” 커밋은 대부분 다음과 같아요:
- 변수명 변경
- 테스트 수정
- 주석 수정
- 스타일 정리
- 빌드 오류 수정

이 변경들은 기존 커밋의 범위 안에 포함돼요.  
따라서 새로운 커밋으로 추가하지 않고, 기존 커밋에 포함시키는 것이 맞아요.

또한, 빌드 오류 수정이 따로 필요하다는 것은  
해당 커밋이 “항상 배포 가능한 상태”가 아니었다는 의미예요.
